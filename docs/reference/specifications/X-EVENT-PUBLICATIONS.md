# x-event-publications

> **Version**: 1.0
> **Status**: Implemented
> **Last Updated**: 2026-03-16
> **Schema Scope**: `*-service-events.yaml`
> **Generated Output**: `{Service}PublishedTopics.cs` (const string topic fields), `{Service}EventPublisher.cs` (typed `Publish*Async` extension methods on `IMessageBus`)
> **Related Specifications**: [x-event-subscriptions](X-EVENT-SUBSCRIPTIONS.md)
> **Tenet References**: T5 (FOUNDATION), T16 (QUALITY)

---

## Summary

Declares all events a service publishes, serving as the authoritative event catalog for that service. Generates typed topic constants and Publish*Async extension methods on IMessageBus, replacing error-prone inline topic strings with compile-time-safe typed publishers. Supports parameterized topics for dynamic routing keys with typed parameter overloads.

---

## Schema Syntax

### Basic Publication Declaration

Publications are declared under `info.x-event-publications` in the service's events schema:

```yaml
info:
  title: Faction Service Events
  version: 1.0.0
  x-event-subscriptions: []
  x-event-publications:
    - topic: faction.created
      event: FactionCreatedEvent
      description: Published when a new faction is created
    - topic: faction.member.added
      event: FactionMemberAddedEvent
      description: Published when a character joins a faction
```

### Parameterized Topics

Events published to dynamic routing keys use `{placeholder}` syntax in the topic and a `topic-params` array:

```yaml
x-event-publications:
  - topic: "map.{regionId}.{kind}.updated"
    event: MapUpdatedEvent
    description: Published when map data is updated
    topic-params:
      - name: regionId
        type: uuid
      - name: kind
        type: MapKind
```

## Field Reference

### Publication Entry Fields

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `topic` | string | Yes | -- | RabbitMQ routing key. Static string for standard events, or string with `{placeholder}` segments for parameterized topics. |
| `event` | string | Yes | -- | Event model class name. Must match a `components/schemas` entry in the same events schema file. |
| `description` | string | Yes | -- | Human-readable description of when this event is published. Used in generated XML documentation. |
| `topic-params` | array | No | -- | Parameter definitions for dynamic topic placeholders. Required when `topic` contains `{placeholder}` segments. |

### topic-params Entry Fields

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `name` | string | Yes | -- | Parameter name. Must match a `{placeholder}` in the `topic` string exactly. |
| `type` | string | Yes | -- | Parameter type. `string` maps to C# `string`, `uuid` maps to C# `Guid`, any other value is used as the C# type directly (e.g., `MapKind` becomes the enum type). |

## Generated Output

### File: `{Service}PublishedTopics.cs`

Generated by `generate-published-topics.py`. Contains `const string` fields for every declared topic.

```csharp
// <auto-generated>
// This code was generated by generate-published-topics.py from faction-service-events.yaml.
// Do not edit this file manually.
// </auto-generated>

namespace BeyondImmersion.BannouService.Faction;

/// <summary>
/// Published event topic constants for FactionService.
/// Use these constants instead of inline topic strings when calling
/// <c>IMessageBus.TryPublishAsync</c>.
/// </summary>
public static class FactionPublishedTopics
{
    /// <summary>Published when a new faction is created.</summary>
    public const string FactionCreated = "faction.created";

    /// <summary>Published when a character joins a faction.</summary>
    public const string FactionMemberAdded = "faction.member.added";
}
```

For parameterized topics, the const field contains the literal template string (with placeholders):

```csharp
/// <summary>Published when map data is updated.</summary>
public const string MapUpdated = "map.{regionId}.{kind}.updated";
```

### File: `{Service}EventPublisher.cs`

Generated by `generate-event-publishers.py`. Contains typed `Publish*Async` extension methods on `IMessageBus`.

**Standard (non-parameterized) method**:

```csharp
/// <summary>Published when a new faction is created.</summary>
public static Task<bool> PublishFactionCreatedAsync(
    this IMessageBus messageBus,
    FactionCreatedEvent eventData,
    CancellationToken cancellationToken = default)
    => messageBus.TryPublishAsync(FactionPublishedTopics.FactionCreated, eventData, cancellationToken);
```

**Parameterized topic -- two overloads generated**:

1. **Base overload** (literal topic, for cases where the caller has the full topic string):
```csharp
/// <summary>Published when map data is updated.</summary>
public static Task<bool> PublishMapUpdatedAsync(
    this IMessageBus messageBus,
    MapUpdatedEvent eventData,
    CancellationToken cancellationToken = default)
    => messageBus.TryPublishAsync(MappingPublishedTopics.MapUpdated, eventData, cancellationToken);
```

2. **Parameterized overload** (with typed parameters and `[ParameterizedTopic]` attribute):
```csharp
/// <summary>Published when map data is updated. Parameterized overload for dynamic topic resolution.</summary>
[ParameterizedTopic(typeof(Guid), typeof(MapKind))]
public static Task<bool> PublishMapUpdatedAsync(
    this IMessageBus messageBus,
    MapUpdatedEvent eventData,
    Guid regionId,
    MapKind kind,
    CancellationToken cancellationToken = default)
    => messageBus.TryPublishAsync($"map.{regionId}.{kind}.updated", eventData, cancellationToken);
```

### Type Mapping for topic-params

| Schema `type` | C# Type |
|---|---|
| `string` | `string` |
| `uuid` | `Guid` |
| Any other value | Used verbatim as C# type (e.g., `MapKind` becomes the enum type) |

## Runtime Behavior

### Publishing Flow

1. Service code calls `_messageBus.PublishFactionCreatedAsync(eventData)` (generated extension method)
2. The extension method calls `IMessageBus.TryPublishAsync` with the correct const topic and typed event
3. `TryPublishAsync` serializes the event via `BannouJson` and publishes to RabbitMQ

### Parameterized Topic Resolution

For parameterized overloads, the topic string is built via C# string interpolation at call time:

```csharp
// Service code
await _messageBus.PublishMapUpdatedAsync(eventData, regionId, MapKind.Terrain);
// Resolves to: TryPublishAsync("map.{regionId}.Terrain.updated", eventData, ct)
```

### Enforcement

Services MUST use the generated extension methods or topic constants instead of inline topic strings. The structural test `Service_CallsAllGeneratedEventPublishers` validates that services do not bypass generated publishers.

## Structural Tests

| Test Name | Validates |
|---|---|
| `Service_CallsAllGeneratedEventPublishers` | Every service that declares `x-event-publications` uses the generated `Publish*Async` extension methods rather than inline topic strings in `TryPublishAsync` calls |
| `Events_PublishedTopicsWithoutSubscribers` | Informational test that lists published topics with no matching `x-event-subscriptions` across any service (checklist, not a failure) |
| `BatchLifecycleEntities_HaveBatchEventPublications` | Batch lifecycle entities have corresponding batch event publication entries |

## Edge Cases & Restrictions

### Forbidden Patterns

| Restriction | Reason |
|---|---|
| Inline topic strings in service code (`_messageBus.TryPublishAsync("faction.created", ...)`) | Must use generated `Publish*Async` extension methods or `PublishedTopics` constants. Structural test enforces this. |
| Inline `$"topic.{param}"` string interpolation in service code | Must use generated parameterized overload which handles interpolation. |
| Missing `topic-params` when topic contains `{placeholder}` | Generator cannot produce typed parameters without `topic-params` declarations. |
| `topic-params` entry with `name` not matching any `{placeholder}` in topic | Parameter would be unused in the generated interpolation. |
| Publishing events not listed in `x-event-publications` | The publications list is the authoritative registry. Unlisted events will lack generated publishers and topic constants. |

### Scoping Rules

- **Publications are per-service**: Each service's `*-service-events.yaml` declares only that service's publications
- **Both lifecycle and custom events must be listed**: Auto-generated lifecycle events (from `x-lifecycle`) should also appear in `x-event-publications` for complete topic constant and publisher generation
- **Event model must exist in same schema**: The `event` class name must match a `components/schemas` entry in the same `*-service-events.yaml` file (unlike subscriptions which resolve cross-schema)

### Interaction with Other Extension Attributes

- **x-event-subscriptions**: Subscriber services reference topics by the exact strings declared in the publisher's `x-event-publications`. Topic strings must match exactly.
- **x-lifecycle**: Lifecycle events are auto-generated from `x-lifecycle` declarations. These should also appear in `x-event-publications` to ensure topic constants and typed publishers are generated.
- **Topic naming conventions**: Topics must follow Pattern A (`entity.action`) for single-entity services or Pattern C (`service.entity.action`) for multi-entity services. Pattern B (service name embedded via hyphens in entity) is forbidden per QUALITY TENETS.
