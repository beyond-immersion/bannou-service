# x-event-template

> **Version**: 1.0
> **Status**: Implemented
> **Last Updated**: 2026-03-16
> **Schema Scope**: `*-service-events.yaml`
> **Generated Output**: `{Service}EventTemplates.cs` — static class with `EventTemplate` fields and `RegisterAll(IEventTemplateRegistry)` method

---

## Summary

Declares that a service event should have an auto-generated template for use with emit_event ABML actions, enabling behavior documents to publish typed events. The generator produces EventTemplate instances with pre-built payload templates that map event properties to template substitution placeholders. Use when NPC behaviors or god-actors need to emit service events from ABML documents.

---

## Schema Syntax

The `x-event-template` attribute is defined on individual event schema definitions within `*-service-events.yaml` files, as a sibling of `required`, `properties`, and `description`:

```yaml
components:
  schemas:
    QuestObjectiveProgressedEvent:
      allOf:
        - $ref: 'common-events.yaml#/components/schemas/BaseServiceEvent'
      type: object
      additionalProperties: false
      description: Published when objective progress changes
      x-event-template:
        name: quest_objective_progressed
        topic: quest.objective.progressed
      required:
        - eventName
        - questInstanceId
        - questCode
        - objectiveCode
        - currentCount
        - requiredCount
        - isComplete
      properties:
        questInstanceId:
          type: string
          format: uuid
          description: Quest instance ID
        questCode:
          type: string
          description: Quest template code
        objectiveCode:
          type: string
          description: Objective code within the quest
        currentCount:
          type: integer
          description: Current progress count
        requiredCount:
          type: integer
          description: Required count to complete
        isComplete:
          type: boolean
          description: Whether the objective is now complete
```

## Field Reference

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | Template name used by ABML `emit_event` actions. Convention: snake_case matching the event's semantic meaning. |
| `topic` | string | Yes | Event topic (RabbitMQ routing key) the template publishes to. Must follow the standard topic naming convention. |

### PayloadTemplate Generation Rules

The payload template maps event properties to Mustache-style `{{placeholder}}` substitutions. Quoting rules depend on the property's schema type:

| Schema Type | Nullable | Template Format | Example |
|-------------|----------|-----------------|---------|
| `type: string` | No (`nullable: false` or not specified) | Quoted: `""{{field}}""` | `""eventName"": ""{{eventName}}""` |
| `type: string` | Yes (`nullable: true`) | Unquoted: `{{field}}` | `""reason"": {{reason}}` |
| `type: integer` | Any | Unquoted: `{{field}}` | `""currentCount"": {{currentCount}}` |
| `type: number` | Any | Unquoted: `{{field}}` | `""score"": {{score}}` |
| `type: boolean` | Any | Unquoted: `{{field}}` | `""isComplete"": {{isComplete}}` |
| `type: array` | Any | Unquoted: `{{field}}` (pre-serialized JSON) | `""tags"": {{tags}}` |
| `type: object` | Any | Unquoted: `{{field}}` (pre-serialized JSON) | `""metadata"": {{metadata}}` |

Non-nullable strings are quoted because they are always present and always strings. Nullable strings are unquoted because the substitution engine may insert `null` (a JSON literal, not a quoted string). Numeric, boolean, array, and object types are never quoted because JSON represents them without quotes.

### Registration

Services must call `RegisterAll` during startup to make templates available to the Actor runtime:

```csharp
public override async Task OnRunningAsync(CancellationToken cancellationToken)
{
    QuestEventTemplates.RegisterAll(_eventTemplateRegistry);
    // ...
}
```

## Runtime Behavior

When an ABML `emit_event` action executes:

1. The Actor runtime resolves the template by `name` from `IEventTemplateRegistry`
2. The `PayloadTemplate` string is populated by substituting `{{placeholder}}` tokens with actual values from the behavior context
3. The populated JSON string is deserialized to the template's `EventType`
4. The event is published to the template's `Topic` via `IMessageBus`

Templates are registered once at startup and are immutable thereafter. Multiple actor instances share the same template instances concurrently.

```yaml
      type: string

      description: Quest instance ID
    questCode:
      type: string

      type: string
      description: Objective code within the quest

      type: integer
      description: Current progress count

      type: integer
```


## Edge Cases & Restrictions

### Forbidden Combinations

| Restriction | Reason |
|---|---|
| `x-event-template` on a non-event schema | Only event definitions (in `*-service-events.yaml`) support this attribute |
| Duplicate `name` values across the same service | Template names must be unique within a service's `RegisterAll` method |
| `name` with spaces or special characters | Template names are identifiers used in ABML; use snake_case |

### Scoping Rules

- Templates are scoped to their service. Two services may define templates with the same `name` without conflict, as they register with separate `RegisterAll` calls.
- The `topic` field must be a valid event topic for the owning service, following the standard topic naming convention (Pattern A or Pattern C).
- Only properties defined directly on the event schema (not inherited from `BaseServiceEvent`) appear in the `PayloadTemplate`, with the exception of `eventName` which is always included.

### Interaction with Other Extension Attributes

- **x-lifecycle**: Lifecycle events may also have `x-event-template` if they need to be emittable from ABML.
- **x-event-publications**: The topic in `x-event-template` should correspond to a published topic declared in `x-event-publications` for the service.

### Known Limitations

- The generator does not support nested object property expansion in templates. Complex object properties are emitted as single `{{placeholder}}` tokens expecting pre-serialized JSON.
- `$ref` properties inherited from `allOf` base schemas (other than `BaseServiceEvent`) are not automatically included in the payload template.
